0. 용어 정의

플랫폼 전반에서 사용하는 핵심 용어를 정리했습니다. BE 개발자·기획자·현업이 동일한 언어로 소통하기 위한 기준으로 작성했습니다.

회사 (Tenant)
tenant_id · B2B SaaS 멀티 테넌시 단위
B2B SaaS 멀티 테넌시의 최상위 격리 단위. 플랫폼에 가입한 각 회사가 하나의 Tenant.

Tenant마다 데이터·사용자·권한·알림설정·디바이스·회사 코드가 완전히 격리됩니다.

Tenant 간 데이터 연결 방식:
· company_code — 외부 회사 간 수평 연결. CargoItem에 상대 회사 코드를 입력하면 해당 Tenant 대시보드에 읽기 전용 자동 연결.
· parent_tenant_id — 같은 법인 내 본사/지사 수직 계층.
예시: 종근당 = Tenant A, 용마로지스 = Tenant B. 용마로지스가 종근당 화물 운송 시 CargoItem에 종근당 company_code 입력 → 종근당 대시보드에 읽기 전용 연결.
그룹 (Group)
group_id · 트래킹 묶음 · 최대 2단계
독립적으로 실행되는 트래킹 여러 개를 하나의 업무 단위로 묶어 통합 조회·관리하는 선택적 단위. 최대 2단계(루트→자식)로 고정.

Segment와의 차이: 하나의 여정을 순서대로 구간 분리할 때는 Segment. 독립 트래킹 여러 개를 업무 단위로 묶을 때는 Group.
예시: 현대차 수출건 HD01-0441-001 → 컨테이너 3개 분할 선적(트래킹 3건). 같은 계약이므로 Group 1개로 묶어 통합 상태 조회 및 정산 단위로 사용.
트래킹 (Tracking)
핵심 모니터링 단위
플랫폼의 핵심 모니터링 단위. 운송 1건 또는 공간 1개를 감시.

· 운송형 — 출발지에서 도착지까지 이동 단위. 실시간 위치·온도·충격 데이터 수집.
· 공간형 — 창고·냉장 챔버 등 고정 공간 상시 모니터링.

상태 라이프사이클:
운송형 (TRANSPORT)
대기중 → 운송중 → 운송완료
취소 시 → 운송취소
공간형 (SPACE)
대기중 → 측정중 → 측정완료
취소 시 → 측정취소
예시(운송형): 용마로지스 냉장 트럭 서울→부산 운행 1건 = 트래킹 1건.
예시(공간형): 종근당 인천 냉장 창고 온도 상시 감시 = SPACE 트래킹 1건.
세그먼트 (Segment)
경유지 분리 (선택)
하나의 트래킹 내 구간을 분리하는 선택적 단위. 구간별로 독립적인 온도 기준·알림 프리셋·측정대상을 적용해야 할 때 사용.

TRANSPORT type에서만 사용. SPACE type에는 세그먼트 없음.
예시: 서울(출발) → 인천항(경유) → 상하이(도착). 세그먼트 1: 서울~인천항 냉장(-18°C), 세그먼트 2: 인천항~상하이 선내 냉동(-20°C). 각 구간에 다른 측정대상과 알림 프리셋 적용.
측정대상 (MeasurementTarget)
물리 자산 · 17종
디바이스가 부착되는 물리 자산. 차량·컨테이너·팔레트·냉장 챔버 등 17종으로 분류.

· 하나의 측정대상에 복수의 디바이스 동시 부착 가능 (온도·GPS·문열림 센서 병행).
· 하나의 측정대상에 복수의 알림 프리셋 적용 가능.
· 측정대상 ↔ 화물(CargoItem)은 N:M — 여러 화물이 동시에 실릴 수 있고, 하나의 화물이 환적으로 여러 측정대상을 거칠 수 있음.
예시: 용마로지스 냉장 트럭 "25가 3451" = MeasurementTarget 1개. 온도 센서 3개(전단·중단·후단) + GPS 1개 + 문열림 센서 1개 = Device 5개 부착.
디바이스 (Device)
IoT 센서 장비
온도·습도·충격·위치 등 센서 데이터를 수집하는 IoT 장비. 하나의 측정대상에 복수(1:N) 부착 가능.

· 실시간 직접 전송 — 서버에 직접 데이터 전송. 즉각 알림 가능.
· 게이트웨이 중계 전송 — 게이트웨이 장비를 통해 중계 전송. 블루투스 소형 센서 등.
예시: 냉장 트럭 1대에 온도 센서(전단·중단·후단) 3개 + GPS 실시간 기기 1개 = Device 4개. 각 센서 데이터가 독립적으로 수집.
알림 프리셋 (AlertPreset)
임계치 조건 + 수신 대상자 관리
임계치 조건을 묶어 재사용하는 단위. 하나의 프리셋을 여러 측정대상에 적용할 수 있고, 반대로 하나의 측정대상에 여러 프리셋을 동시에 연결할 수 있습니다.

알림 발동 흐름 :
1. 측정대상에 연결된 알림 프리셋 임계치 위반 감지
2. 현재 해당 측정대상에 실려 있는 활성 화물 확인
3-A 화물 없음 → 측정대상 소유 회사 기본 담당자에게 발송
3-B 화물 있음 → 각 화물 등록 시 지정한 수신자에게 독립 발송

스코프: 측정대상 레벨. 동일 측정대상이 여러 트래킹에 걸쳐도 프리셋은 한 번만 적용.
예시: "바이오 냉장 표준" 프리셋 = 온도 2~8°C 이탈 + 문열림 10분 이상. default_recipients: 용마로지스 QA팀. → 트럭에 종근당·알콘 화물이 실린 경우 각 화주사 담당자에게도 추가 발송.
회사 코드 (company_code)
크로스 Tenant 연결 코드 · 자동 생성 후 커스터마이징 가능
모든 Tenant가 보유하는 크로스 Tenant 연결 코드. Tenant 생성 시 플랫폼이 자동 생성(대문자 8자). 어드민이 대문자+숫자 조합으로 커스터마이징 가능. 최대 32자. 전체 유니크.

크로스 Tenant 연결은 CargoItem 레벨에서만 동작합니다. CargoItem에 상대 회사의 company_code를 입력하면 해당 Tenant 대시보드에 읽기 전용 조회가 자동 연결.
⚠️ 입력은 선택 — company_code를 입력하지 않으면 자사 전용. CargoItem 미등록 시 크로스 Tenant 연결 불가.
화물 (CargoItem)
측정대상에 실리는 화물 단위
이번 운송에서 측정대상(운반 수단)에 실리는 화물 단위. 측정대상과 N:M 관계 — 환적으로 여러 운반 수단을 거칠 수 있고, 하나의 운반 수단에 여러 화물이 동시에 실릴 수 있습니다.

활성 화물 판단 기준: 적재 시각 이후이고 하차 시각 이전인 화물을 "현재 적재 중"으로 봅니다. 하차 시각이 미기재면 운송 완료 전까지 유지됩니다.
예시(LCL): 컨테이너 1개에 종근당 의약품 + 알콘 안과용액 혼재 → CargoItem 2개. 임계치 이탈 시 각자 자사 담당자에게 독립 발송.

플랫폼 주 사용 회사(사용자)

플랫폼에 가입하는 각 회사가 하나의 Tenant입니다. 회사 타입 구분 없이 모든 기능을 동일하게 사용합니다. 주요 고객군은 화주사·운송사·3PL·포워더이며, 각 회사가 실제로 어떻게 플랫폼을 활용하는지 정리했습니다.

화주사
제조사 · 유통사 · 수출입 업체
  • 트래킹 구간·디바이스·알림 설계 주체 — 물류에서 갑(甲)의 지위
  • 외부 위탁 시 company_code를 운송사·포워더에 제공 → 조회 연결
  • 모니터링·알림·리포트·클레임 단위 관리 모두 화주 주도
운송사
트럭 · 선사 · 항공사 · 철도
  • 트래킹 등록·디바이스 부착·알림 설정 운영 — 가장 일반적인 등록 주체
  • 운송 등록 시 화주의 company_code 입력 → 화주 대시보드에 자동 연결
3PL
종합물류 · 창고+운송 복합
  • 창고(SPACE) + 운송(TRANSPORT) 트래킹 모두 운영
  • 다수 화주사의 화물을 대리 관리. 화주별 company_code로 데이터 격리
포워더
국제물류 주선
  • LCL·복합운송에서 디바이스 직접 운영 시 트래킹 등록 주체
  • 그 외에는 조회 연결(뷰어) 위치. HBL 단위 CargoItem 관리 가능
트래킹 등록 주체는 디바이스·운송수단을 직접 운영하는 회사입니다. 화주사·운송사·3PL·포워더 모두 등록 주체가 될 수 있습니다. 등록 주체가 상대 회사의 회사 코드를 입력하면 해당 회사 대시보드에 읽기 전용 조회가 연결됩니다.

통합플랫폼 전체 주요 구조 관계

플랫폼을 구성하는 핵심 개념과 그 연결 관계를 한눈에 정리했습니다.

Chain A — 세그먼트 없음 (FCL · 단일 화주)
트래킹 → 측정대상 → CargoItem → 제품마스터
Chain B — 세그먼트 경유 (LCL · 복합운송 · 다화주)
트래킹 → 세그먼트 → 측정대상 → CargoItem → 제품마스터

1. 회사(Tenant) 간 연결 구조

Willog 플랫폼의 최상위 단위는 회사(Tenant)입니다. 회사 간 연결은 회사 코드(company_code)로 외부 회사와 수평 연결하거나, 본사/지사 계층으로 같은 법인 내 조직을 구성합니다.

회사 A — 운송 등록 주체
역할
  • 트래킹·디바이스·측정대상·알림 등록·운영
  • 운송 등록 시 상대 회사의 company_code 입력 → 해당 Tenant에 읽기 전용 연결
회사 B — 조회 연결 수신 주체
역할
  • 자사 company_code가 포함된 CargoItem이 있는 Tracking 기준으로 읽기 전용 조회
  • Tracking의 현재 위치·온도·이벤트 등 운송 컨텍스트 전체 접근

4. 알림 프리셋 & 수신 대상자

알림은 임계치 조건(AlertPreset)과 수신 대상자(AlertRecipient)를 분리해 관리합니다. 임계치는 물리 자산(측정대상)에, 수신 대상자는 화물 소유 관계(CargoItem)에 각각 연결됩니다.

알림 프리셋 (AlertPreset) — 임계치 조건 + 기본 수신자
설정 주체: 측정대상 소유 회사 (운송사·3PL)
적용 범위: 측정대상 단위 (하나의 프리셋을 여러 측정대상에 재사용)

포함 내용:
· 알림 조건: 센서 종류별 임계치 및 지속 시간 (예: 온도 8°C 초과, 문열림 10분 이상)
· 기본 수신자: 화물 미등록 시 알림 받을 담당자 목록 및 수신 채널
예시 — "바이오 냉장 표준" 프리셋:
· 온도 8°C 초과 시 즉시 알림
· 문열림 10분 이상 지속 시 알림
· 기본 수신자: 용마로지스 QA팀 (SMS)
알림 수신자 (AlertRecipient) — 화물별 담당자
설정 주체: 화물(CargoItem) 등록 회사 (화주사 / 운송사 / 3PL)
적용 범위: 화물(CargoItem) 단위 — 화물 하나에 여러 수신자 지정 가능

포함 내용:
· 수신자: 플랫폼 내 사용자 또는 외부 연락처
· 수신 채널: SMS / 이메일 / 앱 푸시 / 카카오
· 알림 범위: 전체 알림 또는 특정 센서(온도·문열림 등) 선택 수신
예시: 종근당 의약품 화물 등록 시
· 종근당 QA팀 → SMS (온도 알림 전용)
· 종근당 물류팀장 → 이메일 (전체 알림)

알림 프리셋 ↔ 측정대상 연결 규칙

알림 프리셋은 측정대상에 자유롭게 붙이고 뗄 수 있습니다. 하나의 프리셋을 여러 측정대상에 적용하거나, 하나의 측정대상에 여러 프리셋을 동시에 연결하는 N:M 구조를 지원합니다.
동일한 측정대상에 같은 프리셋을 중복 연결하는 것은 방지됩니다. 연결 시 담당자와 연결 시각이 기록됩니다.

프리셋은 측정대상 단위로 동작합니다. 동일 측정대상이 여러 트래킹에 동시 연결돼 있어도 프리셋은 측정대상 기준으로 한 번만 발동됩니다.

알림 로그: 발동 시점에 해당 측정대상이 어느 트래킹에 연결돼 있었는지가 함께 기록되어 어느 운송 컨텍스트에서 발생했는지 추적 가능합니다.
하이브리드 알림 수신 구조: AlertPreset에는 항상 default_recipients가 있어 CargoItem 등록 여부와 무관하게 측정대상 소유 회사의 담당자는 알림을 받습니다. CargoItem이 존재하면 거기에 등록된 AlertRecipient(화주사 담당자 등)에게 추가로 발송됩니다. 각 회사는 상대방 수신자를 알 필요 없이 자사 설정만으로 독립적으로 동작합니다.

5. 화물(CargoItem) & 제품 마스터

측정대상(운반 수단)과 화물은 N:M 구조입니다. 하나의 화물이 환적을 거쳐 여러 운반 수단을 옮겨 탈 수 있고, 하나의 운반 수단에 여러 화물이 동시에 실릴 수 있습니다. 화물과 제품 마스터도 N:M 구조 — 하나의 화물에 여러 제품이 포함될 수 있습니다. 화물에는 화주사 연결(회사 코드), 알림 수신자가 함께 등록됩니다.

제품 마스터 (ProductMaster)
품목 기준 정보 — 제품 분류, 규격, 보관 온도 범위 등 품목 자체의 기준 정보. 각 Tenant가 사전 등록. 동일 제품을 여러 화물에 재사용 가능. 화물(CargoItem)과 N:M 관계.
예시: "냉장 연어 200g" — 분류: 신선식품, 보관 온도 0~4°C, 취급 주의 사항: 충격 주의
화물 (CargoItem)
이번 운송의 화물 — 이번 운송에서 측정대상에 실리는 실제 화물. 화주 연결(회사 코드), 적재/하차 시각, 알림 수신자 보유. 제품(ProductMaster)과 N:M 관계 — 하나의 화물에 여러 제품 구성 가능.
예시: 종근당 의약품 화물 1건 — 냉장 백신 30박스 + 상온 정제 20박스 (2종 제품 구성)

측정대상 ↔ 화물(CargoItem) 시간 제한 연결

측정대상 ↔ 화물 연결은 N:M이며, 시간 범위(적재 ~ 하차)로 활성 여부를 판단합니다. 동일 화물이 환적 시 다른 운반 수단으로 이동하거나, 하나의 운반 수단에 여러 화물이 동시에 실릴 수 있습니다. 모든 연결 변경 이력이 시간 기록으로 추적됩니다.

환적 (Transshipment)기존 연결 unloaded_at 기록 → 새 연결 생성 (동일 CargoItem, 새 measurement_target)

7. 권장 구조 및 결정 기준

그룹 · 세그먼트 · 알림 프리셋을 언제 사용해야 하는지 기준을 정리했습니다.

핵심 원칙: 트래킹이 플랫폼의 핵심 단위입니다. 필요할 때만 그룹으로 묶고, 필요할 때만 세그먼트로 분할하세요. 과도한 중첩(3단계 이상 그룹)은 구현 불가 — Group depth 최대 1입니다.
그룹(Group) 사용 기준
여러 트래킹을 하나의 업무 단위로 묶을 필요가 있나요?
그룹 사용 권장
분할 선적(같은 계약, 다른 컨테이너), SPACE+TRANSPORT 혼합 관리, 정기 납품 계약 묶음. order_no 일치 조건 없음.
단일 트래킹으로 충분한 경우
단순 A→B 운송, 단일 창고 모니터링. 그룹 없이 트래킹만 권장.
세그먼트(Segment) 사용 기준
경유지별로 다른 온도 조건이나 측정대상이 필요한가요?
세그먼트 사용 권장
서울→대전 구간 2~8°C / 대전→부산 구간 -20°C처럼 구간별 조건이 다를 때. 경유지별 독립 알림 프리셋.
세그먼트 불필요한 경우
단일 온도 조건으로 전 구간 동일 관리. SPACE 트래킹은 항상 세그먼트 미사용.
알림 프리셋 설계 기준
동일한 알림 규칙을 여러 측정대상에 반복 적용하나요?
알림 프리셋 재사용 권장
같은 고객사 화물에 동일 온도 조건 반복 적용. 프리셋 1개 생성 후 N개 측정대상에 연결. 변경 시 프리셋만 수정하면 전체 반영.
단건 적용
특수 화물 1건에만 적용되는 임시 조건. 재사용 가능성이 없다면 프리셋 생성 없이 적용.
CargoItem 사용 기준
동일 측정대상에 여러 화주의 화물이 혼재하나요?
CargoItem 필수 (LCL / 혼재)
화물마다 company_code 부여 → 화주별 데이터 격리. 세그먼트와 함께 Chain B 구조 사용.
CargoItem 선택 (FCL / 단일 화주)
화주 연결 필요 시 CargoItem 등록. 내부 운송만이면 생략 가능.
