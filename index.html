0. 용어 정의

플랫폼 전반에서 사용하는 핵심 용어를 정리했습니다. BE 개발자·기획자·현업이 동일한 언어로 소통하기 위한 기준으로 작성했습니다.

회사 (Tenant)
tenant_id · 독립 조직 단위
플랫폼의 최상위 회사 단위. 플랫폼에 가입한 각 회사가 하나의 Tenant입니다. Tenant는 운송사(Carrier) 또는 화주사(Shipper) 타입을 가지며, 타입에 따라 사용 가능한 기능이 달라집니다. Tenant 간 데이터는 shipper_code로만 크로스 연결됩니다.
그룹 (Group)
group_id · 트래킹 묶음
여러 트래킹을 하나의 여정으로 묶어 통합 조회·관리하는 선택적 단위. 트래킹은 회사에 직접 귀속되며, 그룹은 필요할 때만 생성합니다. 최대 2단계 중첩. 멀티모달 전체 여정(창고 → 차량 → 선박 등)을 하나의 트래킹 묶음으로 표현할 때 사용합니다.
트래킹 (Tracking)
tracking_id · 핵심 모니터링 단위
플랫폼의 핵심 단위. 운송형(TRANSPORT): 이동하는 운송 단위 — 공간형(SPACE): 고정 공간 (창고·챔버 등). 주문/운행/공간 기준으로 등록합니다.
세그먼트 (Segment)
segment_id · 경유지 분리 (선택)
트래킹 내 경유지별로 구간을 분리합니다. 구간별 독립 온도 집계 및 알림 프리셋 적용 가능. 주로 국내 운송에서 활용.
측정대상 (MeasurementTarget)
target_id · 물리 자산
디바이스가 부착되는 물리 자산. 차량·컨테이너·팔레트·챔버 등 16종. 트래킹과 1:1 / 1:N / N:1 관계. 측정대상디바이스는 항상 1:1.
디바이스 (Device)
device_id · 센서 장비
온도·습도·충격 등 센서 데이터를 수집하는 장비. T1G / S1 / T1P / V2P / GW / APP / T1W 모델. 측정대상에 1:1로 부착됩니다.
알림 프리셋 (AlertPreset)
alert_preset_id · 재사용 알림 규칙 묶음
온도·습도·경로 등 알림 조건을 묶어 재사용하는 단위. 트래킹 또는 세그먼트에 N:M으로 연결. 복수 프리셋 동시 적용 가능.
화주코드 (shipper_code)
크로스 Tenant 연결 코드 · 모든 Tenant 보유
모든 Tenant가 보유하는 크로스 Tenant 연결 코드. 내 회사 화물을 다른 운송사가 운송할 때, 운송사가 트래킹 등록 시 내 shipper_code를 입력하면 내 Tenant 대시보드에 자동 연결됩니다. 운송사 타입도 화주 역할 시 자신의 shipper_code를 제공할 수 있습니다.

플랫폼 주 사용자 구조

플랫폼에 가입하는 각 회사는 하나의 Tenant입니다. Tenant 생성 시 운송사(Carrier) 또는 화주사(Shipper) 타입을 선택하며, 타입에 따라 사용 가능한 기능 범위가 달라집니다.

운송사 (Carrier)
플랫폼 주 사용자
  • 차량·컨테이너·항공기 등 운송수단 보유
  • 측정 대상에 센서 디바이스 부착
  • 트래킹 등록 · 그룹 구성 · 알림 설정
  • 온도·위치 등 실시간 모니터링
화주사 (Shipper)
조회 전용 사용자
  • 운송사에 발주 / 주문을 의뢰
  • 자사 화물의 배송 현황 조회만 가능
  • shipper_code로 크로스 Tenant 연결
  • 트래킹 등록 · 디바이스 설정 불가
화주 겸 운송사
운송사 측면으로 사용
  • 자체 운송 수단을 보유한 제조사·유통사
  • 화주이면서 자사 운송까지 직접 관리
  • → 운송사 관점에서 플랫폼 사용
  • → 운송수단·디바이스 등록 및 전체 기능 이용
핵심 원칙: 모든 Tenant는 고유한 shipper_code를 보유합니다. 트래킹 등록 시 상대 회사의 shipper_code를 입력하면 해당 Tenant에서 자동으로 읽기 전용 조회가 연결됩니다. 운송사 타입도 다른 운송사에게 화물을 위탁하는 경우(화주 역할) 자신의 shipper_code를 제공할 수 있습니다.

1. 회사(Tenant) 간 연결 구조

Willog 플랫폼의 최상위 단위는 회사(Tenant)입니다. 플랫폼에 가입한 각 회사가 하나의 Tenant이며, Tenant는 운송사(Carrier) / 화주사(Shipper) 타입을 가집니다. Tenant 간 데이터 연결은 shipper_code를 통해서만 이루어집니다.

회사 A — 운송사 (Carrier)
역할
  • 트래킹·디바이스·측정대상·알림 등록·운영 주체
  • 제품 마스터 등록 (어드민/매니저)
  • 화주코드 입력 → 화주사에 데이터 연결
사용자 역할
어드민 / 매니저 / 일반 담당자
회사 B — 화주사 (Shipper)
역할
  • 운송사가 연결한 트래킹 데이터 읽기 전용 조회
  • 자체 물류 인프라 보유 시 운송사로도 등록 가능 (이중 역할)
조회 범위
자사 shipper_code 가 연결된 트래킹만 노출
회사 간(크로스 Tenant) 연결 케이스
케이스 1 — 사용자 초대
운송사(회사 A)가 화주사 담당자를 회사 A 사용자로 직접 초대합니다.
화주사 담당자는 회사 A 내에서 본인 화주코드에 연결된 트래킹만 조회 가능합니다.
회사 B 없이 운영 가능. 화주사 측 별도 계정 불필요.
케이스 2 — 회사 간 연결
화주사가 별도 회사 B 계정을 독립 운영하면서 운송사(회사 A)에서 shipper_code를 등록하면 회사 B 대시보드에 자동으로 연결된 트래킹이 노출됩니다.
두 회사 계정은 독립적. 데이터는 읽기 전용으로만 공유.

내부 논의 및 검토 필요사항 — 회사(Tenant) 구조

Tenant 구조 설계 과정에서 검토가 필요한 항목들을 정리했습니다.

미확정: 아래 내용은 확정된 스펙이 아닙니다. 팀 논의 후 결론이 나면 해당 항목을 업데이트 예정입니다.

논의 1
Tenant 개념 — 운송사/화주사가 각각 별도 Tenant를 갖는 것이 아니다
기존 오해
운송사 Tenant / 화주사 Tenant가 별개의 단위처럼 표현되어, 두 타입이 구조적으로 분리된 것으로 오해 가능
올바른 개념
회사 = Tenant. 운송사/화주사는 Tenant의 타입(type) 속성값이지, 별개의 구조 단위가 아님
적용
Tenant 생성 시 운송사 / 화주사 타입 중 하나를 선택. 타입에 따라 사용 가능한 기능 범위가 달라짐
논의 2
타입 구분 기준 — "직접 트래킹을 생성·관리하는가?"
회사 타입이 제어하는 것은 기능 접근 범위가 아니라, 데이터 연계 구조(shipper_code)입니다. 타입을 2가지로 단순화한 근거:
운송사 (Carrier) 타입
트래킹을 직접 생성·관리하는 주체.
운송수단·디바이스 등록 가능.
shipper_code 입력하여 화주사와 데이터 연결.
전문 운송사
화주이나 자체 운송수단 보유한 회사
쿠팡처럼 자사 창고·물류센터 운영 회사
화주사 (Shipper) 타입
트래킹을 직접 생성하지 않는 주체.
자사 화물에 연결된 트래킹 조회 전용.
자신의 shipper_code를 운송사에게 제공.
운송수단 없이 외부 운송사에만 의뢰하는 제조사·유통사
논의 3
shipper_code — 모든 Tenant가 보유
결정 근거
운송사 타입도 다른 운송사에게 화물을 위탁하는 경우(화주 역할) 자신의 shipper_code를 제공해야 하므로, 타입에 관계없이 모든 Tenant가 shipper_code를 보유해야 함. 초기 설계부터 커버.
활용 방식
화주사 타입 — shipper_code를 운송사에게 제공. 연결된 트래킹 조회.

운송사 타입 — 평소엔 상대 shipper_code를 입력하는 쪽. 화주 역할 시 자신의 코드 제공.
논의 4
엣지 케이스 — 어떤 타입으로 가입해야 하나?
케이스 타입 이유
화주사인데 자체 운송수단 보유 운송사 직접 트래킹을 생성·관리하는 주체이므로
쿠팡처럼 자사 창고센터 운영 운송사 공간형 트래킹을 직접 등록·관리하므로
운송은 외부 위탁, 조회만 필요한 제조사 화주사 트래킹 생성 없이 자사 화물 조회만 필요
3PL — 타사 화물도 운송, 자사 창고도 운영 운송사 트래킹 직접 생성·관리가 핵심 업무이므로

2. 그룹 / 트래킹 / 세그먼트

플랫폼의 운영 핵심은 트래킹입니다. 트래킹은 회사에 직접 귀속되는 기본 단위이며, 필요 시 그룹으로 묶거나 세그먼트로 경유지를 분리합니다.

트래킹 단위 — 등록 기준은 운영 방식에 따라 결정됩니다
트래킹은 "무엇을 1건으로 볼 것인가"를 운영 주체가 정의하는 단위입니다.
주문 기준, 배송 기준, 운송 기준, 발주 기준, 공간 기준 등 고객사마다 다른 업무 단위로 등록될 수 있습니다. 플랫폼은 이 모든 경우를 동일한 구조(트래킹)로 수용합니다.
주문 기준
주문 1건 = 트래킹 1건
배송 기준
배송 1회 = 트래킹 1건
운송 기준
운송 단위 = 트래킹 1건
발주 기준
발주 1건 = 트래킹 1건
공간 기준
고정 공간 = 트래킹 1건
등록 기준 결정 팁: 같은 화물이라도 "주문별 추적이 필요하면" 주문 기준, "차량 효율 관리가 목적이면" 운행 기준, "공간 상시 관리가 목적이면" 공간 기준으로 등록합니다. 하나의 Tenant 내에서 여러 기준을 혼용할 수 있습니다.
그룹 — 복합 여정 묶음
그룹은 전체 여정을 하나로 묶는 논리 단위입니다.

4. 알림 프리셋

알림 프리셋은 반복 사용 가능한 알림 규칙 묶음입니다. 미리 정의해 두고 여러 트래킹에 N:M으로 연결할 수 있습니다. 세그먼트별로 다른 프리셋 적용도 가능합니다.

(or 세그먼트)
적용 단위
N : M
연결

(규칙 묶음)
재사용 가능
1 : N
포함
알림 규칙
(조건 항목)
임계값 + 수신자
세그먼트별 알림 프리셋: 단일 트래킹 내에서도 경유지(세그먼트)마다 다른 온도 범위를 적용할 수 있습니다. 예: 서울→대전 구간 2~8°C, 대전→부산 구간 -20°C 이하 — 알림 프리셋 별도 연결.

7. 권장 구조 및 결정 기준

그룹 · 세그먼트 · 알림 프리셋을 언제 사용해야 하는지 기준을 정리했습니다.

핵심 원칙: 트래킹이 플랫폼의 핵심 단위입니다. 필요할 때만 그룹으로 묶고, 필요할 때만 세그먼트로 분할하세요. 과도한 중첩(3단계 이상 그룹)은 관리 복잡도를 높이므로 권장하지 않습니다.
그룹(Group) 사용 기준
여러 트래킹을 하나의 여정으로 묶을 필요가 있나요?
그룹 사용 권장
멀티모달(창고→차량→선박→항공) 전체 여정을 단일 그룹으로 표현. 복수 트래킹의 통합 대시보드가 필요할 때. 최대 2단계 중첩까지 허용.
단일 트래킹으로 충분한 경우
단순 A→B 운송, 단일 창고 모니터링. 그룹 없이 트래킹만으로 운영 권장.
세그먼트(Segment) 사용 기준
경유지별로 다른 온도 조건이나 알림 프리셋이 필요한가요?
세그먼트 사용 권장
서울→대전 구간 2~8°C / 대전→부산 구간 −20°C 이하처럼 구간별 온도 기준이 다를 때. 경유지 단위로 독립 알림 프리셋 적용. 주로 국내 운송 복수 경유지 사용.
세그먼트 불필요한 경우
단일 온도 조건으로 전 구간 동일 관리. 경유지가 없거나 조건이 동일하면 세그먼트 없이 트래킹 단위로 운영.
알림 프리셋 설계 기준
동일한 알림 규칙을 여러 트래킹에 반복 적용하나요?
알림 프리셋 재사용 권장
같은 고객사 화물에 동일한 온도 조건을 반복 적용. 프리셋 1개 생성 후 N개 트래킹에 연결. 변경 시 프리셋 하나만 수정하면 전체 반영.
단건 트래킹에 직접 설정
특수 화물 1건에만 적용되는 임시 조건. 재사용 가능성이 없다면 프리셋 생성 없이 트래킹 직접 설정.
측정대상 연결 구조 선택
트래킹과 측정대상의 관계는?
3가지 연결 패턴
1:1 — 트럭 1대에 T1G 1개 (일반 차량)
1:N — 컨테이너 1개에 T1P + V2P 복수 부착
N:1 — 여러 팔레트를 1개 T1G로 통합 모니터링
측정대상디바이스는 항상 1:1
측정대상 하나에 디바이스는 반드시 1개만 연결. 디바이스 교체 시 기존 연결 해제 후 신규 디바이스 재연결.
